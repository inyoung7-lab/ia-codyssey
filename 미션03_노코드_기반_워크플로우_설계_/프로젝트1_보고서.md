# 1. 프로젝트 개요

## 1.1 프로젝트 목적

본 프로젝트의 목적은 Google Sheets에 입력된 지원자 정보를 점수 기준으로 판정하고, 판정 결과에 따라 데이터를 분류하며 안내 메일을 자동으로 발송하는 흐름을 Make와 Zapier에서 구현하여 두 플랫폼의 구성 방식과 사용성을 비교하는 데 있다. 동일한 업무 문제를 서로 다른 노코드 플랫폼으로 다룸으로써 시각적 분기, 조건 설정, 데이터 매핑, 테스트 및 결과 확인 방식의 차이를 검토하였다.

## 1.2 자동화 흐름

목표 흐름은 “Google Sheets 신규 지원자 데이터 입력 → Google Sheets Trigger → 점수 조건 확인 → 결과 시트에 분류 저장 → Gmail 결과 안내”이다. Make에서는 합격자·불합격자 시트 저장과 Gmail 발송을 하나의 시나리오에 통합하였다. Zapier에서는 플랜 환경의 분기 구성 제약을 고려하여 합격용 Zap 1과 불합격용 Zap 2를 별도로 만들고, 각 Zap을 `Google Sheets Trigger → Filter by Zapier → Create Spreadsheet Row → Gmail`의 4단계로 구성하였다([그림 11], [그림 16]).

## 1.3 사용 서비스와 계정

사용 서비스는 Google Sheets, Gmail, Make, Zapier이다. 자동화 테스트에는 `dlsdud8878@gmail.com` 계정을 활용하였으며, Google Sheets에 입력된 지원자 정보가 조건 판정과 메일 본문의 동적 데이터로 사용되었다([그림 6], [그림 9], [그림 18], [그림 20]).

## 1.4 Google Sheets 구조

스프레드시트 문서명은 `지원자 평가 폼(응답)`이다. 원본 데이터가 보이는 탭은 화면 하단에 `설문지 응답 시트1`로 표시되며, 표 영역에는 `Form_Responses`라는 표시도 보인다([그림 1]). 원본 열은 `타임스탬프`, `이름`, `이메일`, `점수`, `지원분야`의 다섯 개다. Make 테스트에는 홍길동의 85점과 60점 데이터를 사용하였다. Zapier 보완 테스트에서는 홍길동·AI·92점 데이터와 김길동·백엔드·52점 데이터를 추가하여 합격과 불합격 흐름을 각각 검증하였다([그림 21]).

결과 탭은 `합격자`와 `불합격자`이며 두 탭의 열 구조는 `이름`, `이메일`, `점수`, `지원분야`다([그림 7], [그림 8]). 합격자 시트에는 85점 데이터 두 건, 불합격자 시트에는 60점 데이터 한 건이 확인된다. 결과 시트에는 타임스탬프 열이 없다.

## 1.5 조건 분기 기준과 최종 결과

Make 합격 경로는 점수 필드에 Numeric operator의 “Greater than or equal to”를 적용하고 기준값을 80으로 설정하였다([그림 3]). 불합격 경로는 “Less than 80”이다([그림 4]). 따라서 80점은 합격 경로에 포함되며 두 조건 사이의 공백이나 중복이 없다.

Zapier 합격용 Zap은 `점수 (Number) Greater than 79`로 설정하여 80점 이상을 합격 대상으로 처리하고([그림 13]), 불합격용 Zap은 `점수 is less than 80`으로 설정하여 80점 미만을 처리하였다([그림 17]). 두 Zap은 동일한 기준값 80을 중심으로 상호 배타적인 조건을 구성한다.

Make에서는 85점 데이터를 합격자 탭에, 60점 데이터를 불합격자 탭에 저장하고 각각의 결과 메일을 발송하였다. Zapier에서는 92점 데이터를 Zap 1에서 합격자로 판정하고, 52점 데이터를 Zap 2에서 불합격자로 판정하였다. 각 Zap은 판정된 데이터를 해당 결과 시트에 저장한 뒤 동적으로 작성한 Gmail 안내 메일을 발송하였다.

## 1.6 자동화 핵심 개념

Trigger는 자동화를 시작하게 만드는 사건이다. 본 프로젝트에서는 Google Sheets에 새로운 지원자 행이 생성되거나 기존 행이 갱신되는 사건이 Trigger에 해당한다. Make는 `Watch New Rows`, Zapier는 `New or Updated Spreadsheet Row`를 사용하였다. Trigger가 실행되면 해당 행의 이름, 이메일, 점수, 지원분야가 이후 단계로 전달된다.

Action은 Trigger 이후 자동으로 수행되는 작업이다. 본 프로젝트의 핵심 Action은 판정 결과를 Google Sheets의 합격자 또는 불합격자 시트에 저장하는 작업과 Gmail로 결과 메일을 발송하는 작업이다. Make는 각 분기마다 `Add a Row`와 `Send an email`을 연결하였고, Zapier는 각 Zap에 `Create Spreadsheet Row`와 `Send Email`을 연결하였다. 따라서 두 도구 모두 Trigger 1개 이상과 Action 2개 이상을 포함한다.

Filter는 입력 데이터가 특정 조건을 만족할 때만 다음 단계로 전달되도록 제어한다. 점수가 80점 이상이면 합격 경로, 80점 미만이면 불합격 경로가 실행되도록 설정하였다. Router는 하나의 입력 흐름을 여러 경로로 나누는 Make의 분기 모듈이다. Make에서는 Router 뒤에 합격 Filter와 불합격 Filter를 각각 연결하였다. Zapier에서는 동일한 분기 목적을 합격용 Zap 1과 불합격용 Zap 2로 분리하고 각 Zap에 Filter를 배치하였다.

## 1.7 두 도구의 동일 워크플로우 구조

| 단계 | Make | Zapier | 공통 역할 |
|---|---|---|---|
| 1. Trigger | Google Sheets Watch New Rows | Google Sheets New or Updated Spreadsheet Row | 신규·수정 지원자 행 감지 |
| 2. 조건 분기 | Router + 합격·불합격 Filter | 합격용 Zap 1 Filter + 불합격용 Zap 2 Filter | 점수 80을 기준으로 판정 |
| 3. Action 1 | Google Sheets Add a Row | Google Sheets Create Spreadsheet Row | 합격자 또는 불합격자 시트 저장 |
| 4. Action 2 | Gmail Send an email | Gmail Send Email | 지원자에게 결과 메일 발송 |

Make는 하나의 시나리오 안에서 Router로 두 경로를 표현하고, Zapier는 두 개의 독립된 Zap으로 같은 논리 구조를 표현한다. 화면 배치는 다르지만 입력 데이터, 조건 기준, 결과 시트 저장, Gmail 발송이라는 업무 단계는 동일하다.

# 2. Make 구현 과정

## 2.1 전체 시나리오와 Trigger

[그림 2]의 시나리오명은 `Integration Google Sheets`다. 첫 모듈은 Google Sheets의 `Watch New Rows`이며, 신규 행을 감지하는 Trigger 역할을 한다. 화면 하단에는 실행 주기 `Every 15 minutes`가 표시된다. Trigger가 읽은 원본 행의 이름, 이메일, 점수, 지원분야는 이후 Filter, Add a Row, Gmail 모듈의 공통 입력 데이터로 전달된다.

## 2.2 Router 사용 이유

Trigger 다음에는 Router가 배치되어 하나의 입력을 합격과 불합격 두 경로로 나눈다([그림 2]). Router가 필요한 이유는 동일한 지원자 레코드에 대해 점수 조건에 따라 서로 다른 후속 Action 묶음을 실행해야 하기 때문이다. 왼쪽 경로는 `합격`, 오른쪽 경로는 `불합격`이라는 라벨을 갖고 각 경로가 별도의 Google Sheets Add a Row와 Gmail Send an email로 연결된다. 이 구조는 한 시나리오 안에서 상호 배타적인 두 처리 흐름을 동시에 시각화한다.

## 2.3 Filter 설정

합격 Filter는 입력 모듈 2번의 `점수 (D)`를 조건 필드로 사용하고 `Greater than or equal to 80`을 설정하였다. 라벨은 `합격`이며 fallback route는 No다([그림 3]). 불합격 Filter도 같은 `점수 (D)`를 사용하되 `Less than 80`으로 설정하고 라벨을 `불합격`으로 지정하였다([그림 4]). 두 필터는 Router가 복제한 데이터 중 해당 조건을 만족하는 데이터만 다음 모듈로 전달한다. 80점을 한 경계로 두 조건이 서로 겹치지 않게 구성되었다.

## 2.4 Google Sheets Add a Row

각 분기에는 Google Sheets `Add a Row` 모듈이 하나씩 존재한다([그림 2]). 합격 경로는 합격자 시트, 불합격 경로는 불합격자 시트를 대상으로 하며 `이름`, `이메일`, `점수`, `지원분야`를 각 열에 대응시킨다. 그 결과 85점 데이터는 합격자 시트에, 60점 데이터는 불합격자 시트에 기록되었다([그림 7], [그림 8]).

## 2.5 Gmail 설정과 동적 본문

합격 Gmail 모듈의 To에는 Trigger에서 가져온 `이메일 (C)`가 동적으로 매핑되어 있다. 제목은 `[지원 결과] 합격 안내`, Body type은 `Raw HTML`이다([그림 5]). 본문에는 고정 문장과 `이름 (B)`, `지원분야 (E)`, `점수 (D)` 토큰을 조합하여 “안녕하세요. [이름] 님 / 지원 분야: [지원분야] / 점수: [점수] / 축하드립니다. / 합격하셨습니다.”가 생성되도록 하였다. 지원자마다 이름, 지원분야, 점수가 자동 치환되는 구조다.

불합격 Gmail 모듈도 To에 `이메일 (C)`를 매핑하고 제목을 `[지원 결과] 불합격 안내`, Body type을 `Raw HTML`로 설정하였다([그림 6]). 본문은 이름·지원분야·점수를 동적으로 삽입한 뒤 “아쉽게도 이번에는 불합격입니다.”라는 고정 문장을 붙인다. 실제 수신 메일은 “아쉽게도 이번에는 불합격입니다.”로 표시되어 설정 내용과 일치한다([그림 10]).

## 2.6 실행 테스트와 실제 결과

[그림 2]의 실행 표시와 결과 화면을 통해 각 모듈의 처리를 확인하였다. 85점 홍길동 데이터는 합격자 탭에 저장되었고([그림 7]), Gmail에는 제목 `[지원 결과] 합격 안내`, 지원분야 AI, 점수 85, 합격 문구가 포함된 메일이 도착하였다([그림 9]). 60점 데이터는 불합격자 탭에 저장되었고([그림 8]), 제목 `[지원 결과] 불합격 안내`, AI, 60점, 불합격 문구가 포함된 메일이 도착하였다([그림 10]).

입력 시트의 점수, 결과 시트의 분류, Gmail 본문의 이름·지원분야·점수가 서로 일치하였다. 이를 통해 Make에서 양방향 조건 분기, 결과 저장, 안내 메일 발송이 순서대로 수행되었음을 검증하였다.

# 3. Zapier 구현 과정

## 3.1 구현 구조

Zapier에서는 프로젝트 수행 당시 플랜의 분기 구성 제약을 고려하여 합격용 Zap 1과 불합격용 Zap 2를 각각 구성하였다. 두 Zap은 모두 `1. Google Sheets — New or Updated Spreadsheet Row`, `2. Filter by Zapier — Filter conditions`, `3. Google Sheets — Create Spreadsheet Row`, `4. Gmail — Send Email`의 직선형 4단계다([그림 11], [그림 16]). 하나의 Zap 안에서 양방향으로 분기하는 대신 조건별 Zap을 독립적으로 구성함으로써 각 흐름의 목적과 실행 결과를 명확히 구분하였다.

## 3.2 Google Sheets Trigger

합격용 Zap의 Trigger 상세 화면에서 Spreadsheet는 `지원자 평가 폼(응답)`, Worksheet는 `설문지 응답 시트1`, Trigger column은 `타임스탬프`로 설정하였다([그림 12]). 이벤트는 `New or Updated Spreadsheet Row`이며 Setup, Configure, Test 단계를 순서대로 완료하였다. 불합격용 Zap도 동일한 Google Sheets 이벤트를 시작점으로 사용하였다.

## 3.3 합격 Filter

합격용 Filter는 `점수`, `(Number) Greater than`, `79`로 구성하였다([그림 13]). 테스트 레코드 60점에 대해 Zap이 계속 실행되지 않는다는 메시지가 표시되었으며, 이는 80점 미만 데이터를 합격 메일 단계로 전달하지 않는 필터의 정상 동작을 보여 준다.

## 3.4 합격자 시트 저장, Gmail 매핑과 테스트

Filter를 통과한 데이터는 세 번째 단계인 `Create Spreadsheet Row`로 전달된다. Spreadsheet는 `지원자 평가 폼(응답)`, Worksheet는 `합격자`로 설정하였으며 이름, 이메일, 점수, 지원분야를 Trigger의 동일 필드와 매핑하였다([그림 22]). 이후 Gmail 제목은 `[지원 결과] 합격 안내`, Body type은 Plain으로 설정하였다([그림 14]). 본문에는 Trigger의 이름, 지원분야, 점수가 토큰으로 삽입되고 “축하드립니다.”와 “합격하셨습니다.”가 고정 문장으로 입력된다.

실행 이력에는 60점 데이터가 합격 Filter에서 중단된 기록과 성공 실행 기록이 함께 표시되어 조건 판정이 의도대로 작동했음을 보여 준다([그림 23]). 실제 합격 테스트에서는 원본 시트의 홍길동·AI·92점 정보가 메일 본문에 반영되어 수신함에 도착하였다([그림 15]). 이를 통해 합격 조건 판정, 결과 시트 Action, 개인화 메일 발송의 연속 흐름을 검증하였다.

## 3.5 불합격자 시트 저장, Gmail 매핑과 테스트

불합격용 Filter는 점수에 `is less than 80`을 적용한다([그림 17]). Filter를 통과한 데이터는 세 번째 단계인 `Create Spreadsheet Row`로 전달된다. Spreadsheet는 `지원자 평가 폼(응답)`, Worksheet는 `불합격자`로 설정하였으며 이름, 이메일, 점수, 지원분야를 Trigger 필드와 매핑하였다([그림 16]). Gmail To에는 Trigger의 이메일 값을 매핑하고, 제목은 `[지원 결과] 불합격 안내`, Body type은 plain으로 설정하였다([그림 18]). 본문에는 이름, 지원분야, 점수를 동적으로 삽입하고 불합격 안내와 지원에 대한 감사 문장을 포함하였다.

[그림 19]의 실행 화면에는 Trigger가 신규 행을 찾고 60점 데이터가 불합격 Filter 규칙을 통과한 결과가 표시된다. 보완 테스트에서는 김길동·백엔드·52점 데이터를 사용하였으며, 해당 값이 동적으로 삽입된 불합격 메일이 수신함에 도착하였다([그림 20]). 이를 통해 불합격 조건 판정, 불합격자 시트 저장 Action, 개인화 메일 전송 흐름을 확인하였다.

# 5. 장단점

## 5.1 Make의 장점

1. Router를 통해 합격과 불합격 경로가 하나의 시나리오에 동시에 나타나 전체 업무 구조를 파악하기 쉽다.
2. 각 경로에 Filter, Add a Row, Gmail을 독립적으로 연결하여 저장과 통지를 하나의 분기 안에서 처리한다.
3. `>=80`과 `<80` 연산자를 직접 사용해 경계값 80을 명확히 표현한다.
4. 원본 필드를 초록색 토큰으로 매핑하여 수신자와 본문이 동적으로 작성되는 구조가 화면에서 분명하다.
5. 합격자·불합격자 시트 저장 결과와 두 Gmail 결과가 모두 남아 전체 목표 흐름의 결과 검증이 가능하다.
6. 모듈별 체크와 실행 건수 표시로 어떤 경로의 어느 단계가 처리되었는지 캔버스에서 확인할 수 있다.

## 5.2 Make의 단점

1. Router와 여러 모듈이 한 화면에 배치되어 초기에 구성 요소가 많아 보인다.
2. Watch New Rows의 시작 위치와 실행 주기를 처음 설정할 때 Google Sheets 구조를 함께 이해해야 한다.
3. 두 결과 시트의 동일한 컬럼을 각 경로에 반복 매핑해야 한다.
4. Raw HTML 본문을 사용할 때에는 서식 구조를 함께 이해해야 하므로 Plain 방식보다 설정 요소가 많다.
5. 스케줄 영역에 `Every 15 minutes`가 보이지만 토글 상태와 실제 자동 실행 여부가 명확하지 않아 운영 상태 판독이 어렵다.
6. 모듈 수가 늘어날수록 캔버스가 커지므로 경로별 실행 상태를 순서대로 확인해야 한다.

## 5.3 Zapier의 장점

1. Trigger, Filter, Create Spreadsheet Row, Gmail이 위에서 아래로 배치되어 단일 Zap의 실행 순서가 명확하다.
2. Trigger 설정이 Setup, Configure, Test로 나뉘고 Spreadsheet, Worksheet, Trigger column이 명확히 표시된다.
3. Filter 실패 시 `Your zap would not have continued`와 불일치 샘플값 60을 제시하여 원인 파악이 쉽다.
4. Gmail 테스트 결과에 메시지 ID, Thread ID, 라벨을 제공하여 Action 수행 여부를 구체적으로 확인할 수 있다.
5. Gmail의 이름·지원분야·점수를 시트 필드 토큰으로 삽입하는 방식이 직관적으로 보인다.
6. 합격용과 불합격용 Zap을 분리하여 각 조건과 결과 시트의 목적을 명확히 유지한다.

## 5.4 Zapier의 단점

1. 합격과 불합격을 각각 별도 Zap으로 만들어 공통 Trigger 설정과 경계 조건을 이중 관리해야 한다.
2. 합격용과 불합격용 Zap에 같은 Trigger 및 컬럼 매핑을 각각 설정해야 한다.
3. Filter 테스트와 Gmail Action 테스트가 분리되어 있어 각 테스트의 목적을 구분하여 해석해야 한다.
4. 합격 조건을 `Greater than 79`로 표현하여 정수 점수에서는 맞지만 `>=80`보다 의도가 덜 직접적이다.
5. 두 Zap의 이름이 모두 `Untitled Zap`으로 보여 운영 시 식별성과 유지보수성이 낮다.
6. 무료 플랜의 Action 구성 제약으로 인해 합격용과 불합격용 Zap을 별도로 관리해야 했다.

# 6. 어떤 상황에서 적합한가

아래 추천은 이번 프로젝트에서 경험한 구성 복잡도, 조건 분기 방식, 단계 관리 방식에 근거하여 정리하였다.

| 관점 | Make가 적합한 상황 | Zapier가 적합한 상황 |
|---|---|---|
| 기업 | 한 입력에서 여러 부서·결과로 나뉘고 저장과 통지를 한 시나리오에서 추적할 때 | 짧고 독립적인 업무를 단계별로 빠르게 구성할 때 |
| 개인 | 하나의 자동화 안에서 합격·불합격처럼 여러 분기를 관리하려 할 때 | Google Sheets에서 조건 확인 후 메일을 보내는 단순 자동화가 필요할 때 |
| 학생 | Router, Filter, 데이터 매핑 등 자동화 구조를 시각적으로 학습할 때 | Setup·Configure·Test 순서로 기본 개념을 익힐 때 |
| 스타트업 | 요구사항 변화로 분기와 후속 Action이 늘어날 가능성이 있을 때 | 초기 검증용 단일 흐름을 짧게 만들 때 |
| 중소기업 | 한 담당자가 하나의 캔버스에서 결과 저장과 메일 발송을 함께 유지할 때 | 업무별로 Zap을 분리해 단순한 단위로 담당할 때 |
| 대기업 | 다수의 분기와 후속 작업을 하나의 시각적 구조로 표준화할 때 | 부서별로 독립적인 단순 자동화를 구분하여 관리할 때 |

이번 프로젝트에서는 결과 시트 분기 저장과 메일까지 한 시나리오에서 처리한 Make가 통합 자동화에 더 적합했다. Zapier는 합격용과 불합격용 Zap을 분리하여 각 Zap 안에서 조건 판정, 결과 시트 저장, 메일 발송의 순서를 직선적으로 유지할 수 있었다.

## 6.1 특정 업무에 대한 도구 선택

본 프로젝트와 같이 하나의 입력을 두 개 이상의 조건 경로로 나누고 각 경로에서 시트 저장과 메일 발송을 연속 수행하는 업무에는 Make를 우선 선택하는 것이 적합하다. Router를 사용하면 합격과 불합격 경로를 한 화면에서 비교할 수 있고, 공통 Trigger를 한 번만 관리하면서 각 분기의 Action을 확장할 수 있기 때문이다.

반대로 승인 알림, 정기 안내, 특정 조건의 단일 메일 발송처럼 실행 순서가 짧고 직선적인 업무에는 Zapier가 적합하다. Setup, Configure, Test로 이어지는 단계별 화면과 상세 실행 이력을 통해 각 Action을 순서대로 확인하기 쉽기 때문이다. 따라서 도구 선택은 단순한 기능 수보다 분기 복잡도, 유지보수 방식, 운영자가 선호하는 화면 구조를 기준으로 결정해야 한다.

| Make | 불합격 경로 | 홍길동, AI, 60점 | 불합격자 시트 저장 및 불합격 메일 수신 | [그림 8], [그림 10] |

| Zapier | Zap 2 불합격 경로 | 김길동, 백엔드, 52점 | 불합격 Filter 통과, 불합격자 시트 Action 실행, 불합격 메일 수신 | [그림 16], [그림 19], [그림 20], [그림 21] |

두 도구 모두 합격과 불합격 경로를 각각 1회 이상 테스트하였다. Make는 하나의 Router에서 두 경로를 실행하였고, Zapier는 동일한 두 경로를 Zap 1과 Zap 2로 나누어 실행하였다. 입력 점수와 지원자 정보가 결과 시트 Action 및 Gmail 본문에 동적으로 전달되었으므로 단순한 화면 설계가 아니라 실제 데이터 기반 자동화가 동작한 것으로 확인하였다.

# 8. 프로젝트를 진행하며 느낀 점

이번 프로젝트를 통해 노코드 자동화는 단순히 서비스를 연결하는 작업이 아니라, 입력 데이터의 구조와 조건 경계, 실행 경로, 결과 검증을 함께 설계하는 과정이라는 점을 확인했다. 특히 80점을 기준으로 합격과 불합격을 나누는 작업에서는 연산자를 정확히 선택하고 두 조건이 겹치거나 빠지지 않도록 구성하는 일이 중요했다. Make는 Router 한 개에서 두 경로를 펼치고 각 경로에 시트 저장과 메일 발송을 연결할 수 있어 전체 업무를 한 화면에서 이해하기 좋았다. 반면 모듈 수가 늘어날수록 캔버스 구조와 각 모듈의 데이터 매핑을 체계적으로 관리해야 한다는 어려움도 있었다.

Zapier는 Trigger, Filter, Create Spreadsheet Row, Gmail을 Setup·Configure·Test 순서로 설정하여 개별 단계의 상태를 확인하기 쉬웠다. Filter 테스트는 조건에 맞지 않는 데이터가 다음 단계로 진행되지 않는 것을 보여 주었고, 실행 이력은 성공과 필터 중단을 구분해 표시하였다. 합격과 불합격을 별도 Zap으로 구성하면서 같은 컬럼을 두 번 매핑해야 했지만, 각 흐름이 직선형으로 정리되어 단계별 목적을 파악하기 쉬웠다. 이 과정에서 플랫폼 제약을 실패 요인이 아니라 워크플로우 구조를 결정하는 설계 조건으로 해석하는 방법을 배웠다.

자동화의 장점은 반복 판정, 데이터 분류, 안내 메일 발송을 일관된 규칙으로 처리하여 수작업 누락을 줄일 수 있다는 데 있다. 채용뿐 아니라 교육 성적 통보, 신청 승인, 고객 등급 분류 등에도 같은 패턴을 적용할 수 있다. 향후에는 Zap 이름을 `합격자 처리`와 `불합격자 처리`처럼 명확히 지정하고, 경계값 80을 포함한 테스트 데이터와 실행 기록의 관리 기준을 표준화하여 유지보수성을 높일 수 있다.

# 9. 결론

본 프로젝트는 Google Sheets의 지원자 데이터를 점수 80을 기준으로 분류하고 결과 시트 저장과 Gmail 통지를 자동화하는 것을 목표로 하였다. Make에서는 Watch New Rows, Router, 두 Filter, 두 Add a Row, 두 Gmail 모듈로 목표 구조를 한 시나리오에 구현하였다. 85점 데이터가 합격자 시트와 합격 메일로, 60점 데이터가 불합격자 시트와 불합격 메일로 처리된 결과가 확인되므로 제공 자료 기준으로 전체 목표를 달성하였다.

Zapier에서는 프로젝트 수행 당시 플랜의 분기 구성 제약을 고려하여 합격용과 불합격용 두 개의 4단계 Zap으로 분리하였다. 각 Zap은 Google Sheets Trigger, Filter, Create Spreadsheet Row, Gmail로 구성하였다. 합격용 Zap은 80점 이상 데이터를 합격자 시트에 저장하고 합격 메일을 발송하며, 불합격용 Zap은 80점 미만 데이터를 불합격자 시트에 저장하고 불합격 메일을 발송한다. 92점 합격 사례와 52점 불합격 사례를 통해 두 흐름을 검증하였다.

객관적으로 본 프로젝트의 통합 자동화에는 Make가 더 적합했다. Router를 사용하여 양방향 조건 분기, 결과 시트 저장, 메일 발송을 하나의 시나리오에서 관리할 수 있었기 때문이다. Zapier는 직선형 단계 구성과 구체적인 테스트 메시지가 강점이었으며, 합격용과 불합격용 Zap을 분리해 각 흐름의 조건 판정, 시트 저장, 메일 발송을 독립적으로 관리하는 데 적합했다. 결과적으로 Make는 하나의 캔버스에서 관리하는 복합 워크플로우, Zapier는 목적별로 분리된 직선형 워크플로우에서 각각 장점을 보였다.
